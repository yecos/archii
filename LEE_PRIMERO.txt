PASOS OBLIGATORIOS AL INICIAR CUALQUIER SESION
================================================================================

  1. LEER este archivo completo (LEE_PRIMERO.txt)
  2. LEER INSTRUCTIVO_BITACORA.txt completo (es la fuente de verdad)
  3. LEER worklog.md para ver que hicieron las sesiones anteriores
  4. EJECUTAR: git pull origin main
  5. REVISAR la bitacora (PARTE 4 del instructivo) para entender el estado actual
  6. NUNCA asumir que algo funciona — verificar en el codigo
  7. NUNCA escribir codigo sin entender el estado actual del proyecto

  SI NO CUMPLES ESTOS PASOS, PUEDES CAUSAR:
  - Perdida de datos de usuarios
  - Romper la app en produccion
  - Sobreescribir trabajo de otras sesiones
  - Errores criticos que toman horas resolver

================================================================================
REGLAS CRITICAS DE TRABAJO CONCURRENTE
================================================================================

  [LOCK] ANTES DE TOCAR CUALQUIER ARCHIVO:
  ├── Hacer git pull origin main PRIMERO
  ├── Si el archivo fue modificado por otra sesion, NO sobreescribir
  ├── Leer el archivo actual ANTES de modificarlo
  └── Si hay conflictos: avisar al usuario y NO resolver con force push

  [SAFE] PROTECCION DE DATOS:
  ├── NUNCA eliminar datos de Firestore sin confirmacion del usuario
  ├── NUNCA modificar el array 'members' de un tenant sin saber que contiene
  ├── NUNCA tocar la coleccion 'users' directamente (solo lectura)
  ├── Siempre hacer tag de respaldo antes de cambios grandes:
  │     git tag -a "backup-pre-[descripcion]-[fecha]" -m "Respaldo"
  │     git push origin --tags
  └── Commits cada 15-30 min + push cada hora

  [COMM] COMUNICACION ENTRE SESIONES:
  ├── Toda sesion DEBE registrar su trabajo en INSTRUCTIVO_BITACORA.txt
  ├── Toda sesion DEBE registrar su trabajo en worklog.md
  ├── Documentar: que se hizo, que archivos se tocaron, que commit se hizo
  ├── Al finalizar: commit + push OBLIGATORIO
  └── NUNCA cerrar sesion sin documentar

  [DEPLOY] PROTECCION DE PRODUCCION:
  ├── NUNCA hacer push a main sin verificar que build pasa localmente
  ├── Ejecutar: npx next lint && npx next build (o bun equivalent)
  ├── Vercel tiene limite de 100 deploys/dia — no desperdiciar
  ├── Si algo se rompe en produccion: hacer hotfix inmediato
  └── Documentar todo incidente en la bitacora
