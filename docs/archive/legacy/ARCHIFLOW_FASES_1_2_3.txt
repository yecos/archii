FASE 1: QUICK WINS (2-4 SEMANAS)
--------------------------------
OBJETIVO: Offline básico, virtualización de listas, audit logs simples, feature flags, CI/CD GitHub Actions.

PASO 1.1: OFFLINE-FIRST BÁSICO
- Instalar: npm i idb
- Crear src/lib/offline-queue.ts
- Patrón: escribir en IndexedDB si navigator.onLine === false
- Sync automático: window.addEventListener("online", syncOfflineQueue)
- Validación: Desconectar red → crear tarea → reconectar → verificar persistencia en Firestore

PASO 1.2: VIRTUALIZACIÓN DE LISTAS
- Instalar: npm i @tanstack/react-virtual
- Reemplazar .map() en listas >50 ítems con useVirtualizer
- Aplicar en: Kanban, Timeline, Notificaciones, Lista de proyectos
- Validación: Scroll suave sin lag, memoria estable, DevTools Performance sin warning

PASO 1.3: AUDIT LOGS SIMPLES
- Crear Cloud Function onWrite para colecciones críticas (tasks, files, tenants)
- Escribir en colección audit_logs: { userId, tenantId, action, collection, docId, before, after, timestamp }
- Regla de Firestore: solo admins pueden leer audit_logs
- Validación: Crear/editar/borrar → verificar registro en audit_logs

PASO 1.4: FEATURE FLAGS
- Usar config local o Unleash/Flagsmith gratuito
- Crear src/lib/feature-flags.ts
- Patrón: export const isFlagEnabled = (flag: string) => process.env[`NEXT_PUBLIC_FLAG_${flag}`] === "true"
- Aplicar en rutas nuevas antes de merge
- Validación: Cambiar flag → reiniciar dev → verificar activación/desactivación sin redeploy

PASO 1.5: CI/CD GITHUB ACTIONS
- Crear .github/workflows/ci.yml
- Triggers: push a main, pull_request
- Jobs: npm ci, npx next lint, npx next build, npm test
- Si pasa: auto-deploy a Vercel (manual o automático según preferencia)
- Validación: PR abre → CI corre → falla en lint/build bloquea merge

FASE 2: CORE ENTERPRISE (1-2 MESES)
-----------------------------------
OBJETIVO: IA RAG por tenant, Health Score predictivo, SSO/SAML, API pública + webhooks.

PASO 2.1: IA RAG POR TENANT
- Indexar docs de OneDrive + Firestore en embeddings (Vertex AI o Pinecone)
- Chunking: dividir por párrafos/secciones con metadata { tenantId, source, type }
- Query: vector search + filtrado estricto por tenantId
- Endpoint: /api/ai/rag (Server Action o Route Handler)
- Validación: Preguntar info de tenant A → NO devuelve datos de tenant B

PASO 2.2: HEALTH SCORE PREDICTIVO
- Agregar campos: task completion rate, delay count, resource load, client feedback
- Cloud Scheduler: diario → calcular score 0-100 por proyecto
- Almacenar en project_health_scores collection
- UI: semáforo visual + alertas si score < 60
- Validación: Historial de 30 días → score refleja tendencias reales

PASO 2.3: SSO/SAML + SCIM
- Configurar Firebase Auth → SAML Provider (Azure AD / Okta / Google Workspace)
- Mapear roles de IdP a roles internos (admin, editor, viewer)
- Provisioning: webhooks SCIM para crear/desactivar usuarios automáticamente
- Validación: Login corporativo → tenant correcto → permisos aplicados

PASO 2.4: API PÚBLICA + RATE LIMITING
- Rutas: /api/v1/projects, /api/v1/tasks, /api/v1/files
- Autenticación: API Key por tenant + JWT opcional
- Rate limit: @upstash/ratelimit ou Vercel Edge Middleware
- OpenAPI: swagger.json auto-generado con zod + openapi-fetch
- Validación: Postman/curl → 200 con key válida, 429 con exceso, 403 sin key

PASO 2.5: WEBHOOKS SYSTEM
- Colección webhooks: { tenantId, url, events[], active, secret }
- Event queue: Cloud Tasks / BullMQ → POST con retry exponencial
- Payload estandarizado: { event, tenantId, payload, timestamp, signature }
- Validación: Endpoint prueba recibe eventos → firma HMAC verificada
