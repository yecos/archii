PARTE 2: ARREGLO DE BUGS (UN BUG A LA VEZ)
================================================================================

REGLA: Un bug a la vez. No mezclar correcciones de multiples bugs en un
mismo cambio. Esto permite revision en tiempo real via Vercel preview.

--------------------------------------------------------------------------------
2.1 PASOS
--------------------------------------------------------------------------------

  PASO 1 - IDENTIFICACION DEL BUG
  ├── Registrar el bug en la bitacora con:
  │   - Descripcion del problema
  │   - Pasos para reproducirlo
  │   - Severidad: [CRITICA] / [ALTA] / [MEDIA] / [BAJA]
  │   - Archivos potencialmente afectados
  └── Asignar ID: BUG-[YYYYMMDD]-[NNN]
      Ejemplo: BUG-20260414-001

  PASO 2 - RESPALDO (OBLIGATORIO)
  ├── Tag de respaldo:
  │     git tag -a "backup-pre-bug-[ID]-[YYYYMMDD-HHMM]" -m "Respaldo antes de fix [ID]"
  │     git push origin --tags
  └── Registrar en bitacora

  PASO 3 - CORRECCION
  ├── Modificar UNICAMENTE lo necesario para corregir ESTE bug
  ├── No mezclar con otros bugs ni agregar funcionalidades extras
  ├── Commit descriptivo:
  │     git add [archivo(s)]
  │     git commit -m "fix([ID]): [descripcion de la correccion]"
  │     Ejemplo: fix(BUG-20260414-001): corregir validacion de email en login
  └── Push inmediato:
        git push origin main

  PASO 4 - VERIFICACION EN TIEMPO REAL
  ├── Vercel hace deploy automatico (o preview)
  ├── Verificar que el bug esta corregido
  ├── Verificar que NO se rompio nada existente (regresion)
  ├── Si hay problemas:
  │   - Revertir si es necesario: git revert HEAD
  │   - Corregir y hacer nuevo commit + push
  │   - Verificar de nuevo
  └── Documentar resultado en bitacora

  PASO 5 - CIERRE
  ├── Marcar bug como [RESUELTO] / [PARCIALMENTE RESUELTO] / [NO RESUELTO]
  ├── Si no se resuelve, documentar por que y que se intento
  └── Pasar al siguiente bug (repetir desde PASO 1)

2026-04-20 | FIX — No se ven las personas en la app (Race Condition en teamUsers)
  Estado: COMPLETADO
  Commit: 8644aac
  Build: exitoso (0 errores)

  Contexto: Los usuarios (miembros del equipo) no aparecian en la pantalla de
    Equipo ni en los selectores de asignación. La lista quedaba vacía o mostraba
    datos incorrectos depsues de seleccionar o cambiar de espacio de trabajo.

  Problema identificado — Race Condition:
    El useEffect original de teamUsers combinaba dos operaciones asíncronas en
    un solo efecto:
    1. onSnapshot('users') → escucha todos los usuarios de Firestore
    2. Filtrado con activeTenantMembers → solo mostrar miembros del tenant activo

    El problema: activeTenantMembers se cargaba desde un SEGUNDO useEffect
    que escucha el documento 'tenants/{activeTenantId}'. Cuando los miembros
    del tenant finalmente llegaban, el onSnapshot de 'users' NO se volvía a
    disparar (porque no había cambios en la colección 'users'). Resultado:
    teamUsers quedaba vacía.

    Flujo con bug:
      switchTenant() → activeTenantId = X → activeTenantMembers = []
      → useEffect team ejecuta → setTeamUsers(allUsers) (muestra todos, mal)
      → useEffect tenant members ejecuta → setActiveTenantMembers([uid1, uid2])
      → NINGÚN efecto re-filtra teamUsers → lista rota

  Solución — Separar en 3 efectos independientes:
    1. allUsersCache: onSnapshot('users') → cache de TODOS los usuarios
       - Se ejecuta una vez al iniciar sesión
       - No depende de activeTenantMembers
    2. Tenant members listener: onSnapshot('tenants/{id}') → members[]
       - Carga los UIDs de los miembros del tenant activo
       - Independiente del listener de usuarios
    3. Derivación de teamUsers:
       useEffect(() => {
         if (activeTenantId && activeTenantMembers.length > 0) {
           setTeamUsers(allUsersCache.filter(u => activeTenantMembers.includes(u.id)));
         } else {
           setTeamUsers(allUsersCache);
         }
       }, [allUsersCache, activeTenantId, activeTenantMembers]);

    Ahora cuando activeTenantMembers cambia, el efecto 3 se ejecuta
    inmediatamente y re-filtra el cache de usuarios. Sin race condition.

    src/contexts/AppContext.tsx (+19, -10)

2026-04-20 | FIX — Login con Google y Microsoft (CSP + Headers)
  Estado: COMPLETADO
  Commit: 5c66a22
  Build: exitoso (0 errores)

  Contexto: El login con Google y Microsoft fallaba por restricciones en las
    cabeceras de seguridad (Content Security Policy y X-Frame-Options).
    Los iframes intermedios de Firebase Auth y los endpoints CDN de Microsoft
    no estaban permitidos, causando que los popups OAuth fallaran silenciosamente
    o mostraran errores de "popup blocked" / "unauthorized domain".

  Problemas identificados (4):
    1. X-Frame-Options: DENY bloqueaba iframes intermedios de Firebase Auth
       necesarios durante el flujo OAuth popup
    2. frame-src en CSP faltaba *.firebaseapp.com y *.firebaseio.com
       (Firebase Auth abre iframes intermedios desde estos dominios)
    3. connect-src en CSP faltaba aadcdn.msauth.net y aadcdn.msftauth.net
       (Microsoft Authentication Library carga recursos desde estos CDN)
    4. Permissions-Policy bloqueaba completamente camera, microphone y geolocation
       lo cual podia interferir con flujos OAuth en dispositivos moviles

  Correcciones aplicadas (archivo: next.config.ts):
    * X-Frame-Options: DENY → SAMEORIGIN
      - SAMEORIGIN permite iframes del mismo origen, necesarios para Firebase Auth
    * frame-src: agregados 4 dominios
      - https://*.firebaseapp.com (iframes Firebase Auth intermedios)
      - https://*.firebaseio.com (Firebase Realtime Database)
      - https://login.live.com (Microsoft Live Auth fallback)
    * connect-src: agregados 2 dominios CDN de Microsoft
      - https://aadcdn.msauth.net (Microsoft Authentication Library CDN)
      - https://aadcdn.msftauth.net (Microsoft Authentication CDN alt)
    * Permissions-Policy: camera=(), microphone=(), geolocation=()
      → camera=(self), microphone=(self), geolocation=(self)
      - Permite estos permisos solo desde el mismo origen
    * img-src: agregado https://*.googleapis.com para avatares de Google
    * CSP reorganizado como array join(' ') para legibilidad, con comentarios
      por cada directiva explicando su proposito

  Impacto en seguridad:
    - SAMEORIGIN sigue siendo seguro: no permite que sitios externos incrusten
      la app, pero si permite iframes del propio dominio (Firebase Auth)
    - Los dominios agregados son exclusivamente de Google y Microsoft,
      servicios oficiales requeridos para que OAuth funcione
    - No se aflojan las restricciones de scripts ni estilos

    next.config.ts (+46, -3)

================================================================================
PARTE 5: REGLAS PARA TRABAJO MULTI-SECCION EN EQUIPO
================================================================================

Esta seccion establece las reglas y procedimientos para que multiples
personas o "secciones" (equipos de trabajo) puedan colaborar en el
proyecto ArchiFlow simultaneamente sin conflictos ni perdida de datos.

--------------------------------------------------------------------------------
5.1 ROLES Y RESPONSABILIDADES
--------------------------------------------------------------------------------

  [A] SECCION PRINCIPAL (Lead Developer / Architect)
  ├── Coordina todas las secciones de trabajo
  ├── Aprueba cambios antes de merge a main
  ├── Resuelve conflictos de merge entre secciones
  ├── Mantiene actualizado el INSTRUCTIVO_BITACORA.txt
  ├── Revisa pull requests antes de integrar
  └── Es el punto de contacto para decisiones arquitectonicas

  [B] SECCIONES DE DESARROLLO (Developers)
  ├── Cada seccion trabaja en su rama (branch) propia
  ├── Sigue el instructivo al pie de la letra
  ├── Registra TODOS sus cambios en la bitacora
  ├── No modifica archivos de otra seccion sin coordinacion
  ├── Hace pull de main al menos cada hora de trabajo
  └── Reporta bloqueos inmediatamente al Lead

  [C] SECCION DE QA / TESTING
  ├── Prueba features completadas antes de marcarlas como listas
  ├── Reporta bugs en la bitacora con formato BUG-[fecha]-[NNN]
  ├── No modifica codigo de produccion directamente
  ├── Usa Vercel Preview para pruebas
  └── Documenta pasos de reproduccion detallados

--------------------------------------------------------------------------------
5.2 RAMAS (BRANCHES) POR SECCION
--------------------------------------------------------------------------------

  Estructura obligatoria de ramas:

  main (produccion)
  ├── section/[nombre-seccion]/[feature-o-tarea]
  │   Ejemplos:
  │   ├── section/frontend/chat-whatsapp
  │   ├── section/backend/api-whatsapp
  │   ├── section/fixes/profile-crash
  │   └── section/features/gps-tracking
  └── hotfix/[descripcion-corta]
      (para fixes criticos que no pueden esperar)

  Reglas de ramas:
    1. NUNCA trabajar directamente en main (excepto hotfixes criticos)
    2. Cada seccion crea su rama desde main actualizado:
         git checkout main
         git pull origin main
         git checkout -b section/[seccion]/[tarea]
    3. Nombre de rama debe ser descriptivo y en ingles
    4. Commits frecuentes (cada 15-30 min de trabajo activo)
    5. Push de rama al remoto al menos cada hora:
         git push origin section/[seccion]/[tarea]
    6. Merge a main SOLO via pull request aprobado por Lead

--------------------------------------------------------------------------------
5.3 COORDINACION ENTRE SECCIONES
--------------------------------------------------------------------------------

  [A] ARCHIVOS COMPARTIDOS — ZONA DE CONFLICTO
  ├── Los siguientes archivos son de alta frecuencia de conflicto:
  │   - src/lib/types.ts (tipos globales)
  │   - src/lib/helpers.ts (funciones utilitarias)
  │   - src/contexts/AppContext.tsx (provider principal)
  │   - src/contexts/FirestoreContext.tsx (datos principales)
  │   - src/app/HomeContent.tsx (rutas y navegacion)
  │   - src/lib/firebase-admin.ts (configuracion Firebase)
  │   - INSTRUCTIVO_BITACORA.txt (documentacion)
  │
  ├── REGLA: Antes de tocar un archivo compartido:
  │   1. Hacer git pull origin main
  │   2. Verificar que nadie mas lo esta modificando
  │   3. Comunicar en el canal del equipo: "Voy a modificar [archivo]"
  │   4. Hacer los cambios lo mas atomicos posibles
  │   5. Commitear y push inmediatamente
  │
  └── Si hay conflicto de merge:
      1. NO resolver con git push --force (PROHIBIDO)
      2. Comunicar al Lead inmediatamente
      3. Resolver manualmente conservando ambos cambios
      4. Pedir revision antes de hacer push del merge

  [B] ARCHIVOS POR SECCION — SIN CONFLICTO
  ├── Cada seccion tiene archivos "propios" que no deberian tocar otras:
  │
  │   Pantallas (screens/):
  │   - Cada seccion es duena de sus pantallas asignadas
  │   - No modificar pantallas de otra seccion sin permiso
  │
  │   API Routes (api/):
  │   - Cada endpoint pertenece a la seccion que lo creo
  │   - Modificaciones requieren coordinacion
  │
  │   Contexts (contexts/):
  │   - Contexts compartidos requieren acuerdo previo
  │   - Contexts nuevos se agregan con aviso al equipo
  │
  └── Librerias (lib/):
      - Nuevos archivos lib/ son seguros de crear
      - Modificar lib/ existentes requiere coordinacion

  [C] COMUNICACION OBLIGATORIA
  ├── Antes de iniciar trabajo en un area compartida: AVISAR
  ├── Al terminar trabajo que afecta a otros: NOTIFICAR
  ├── Al encontrar un bug: REGISTRAR en bitacora inmediatamente
  ├── Al bloquearse: PEDIR AYUDA (no perder mas de 15 min)
  └── Al hacer merge a main: CONFIRMAR que build pasa

--------------------------------------------------------------------------------
5.4 FLUJO DE TRABAJO DIARIO (DAILY WORKFLOW)
--------------------------------------------------------------------------------

  INICIO DE SESION (CADA SECCION):
    1. Leer INSTRUCTIVO_BITACORA.txt completo
    2. Hacer git pull origin main
    3. Revisar bitacora para entender estado actual
    4. Checkear canal de equipo por anuncios del Lead
    5. Registrar inicio de sesion en bitacora:
       "2026-MM-DD HH:MM | SECCION [nombre] — Inicio sesion"
    6. Crear/continuar rama de seccion

2026-04-14 15:30 | BUG FIX - Arreglar login con Google, Microsoft y correo/contraseña
  Estado: RESUELTO
  Tag de respaldo: backup-pre-fix-login-auth-20260414
  Objetivo: Diagnosticar y corregir problemas en los 3 metodos de autenticacion
  Commit: ee16a02

  Problemas identificados:
  BUG-20260414-005: Login con Google y Microsoft fallaba silenciosamente
    Severidad: CRITICA
    Causa raiz: En AuthContext.tsx, las funciones doGoogleLogin y doMicrosoftLogin
      accedian a los constructores GoogleAuthProvider y OAuthProvider usando
      (fb as any).auth.GoogleAuthProvider() y (authNS as any).OAuthProvider().
      El casting con 'as any' podria causar problemas en tiempo de ejecucion
      con el SDK compat cargado via CDN. Se cambio a usar una variable
      intermedia const authNS = (fb as any).auth para acceder correctamente
      al namespace que contiene los constructores.

    Adicionalmente, el login de Microsoft solicitaba scopes agresivos
      (Files.ReadWrite.All, Sites.ReadWrite.All) que causaban errores de
      consentimiento (auth/internal-error) cuando el usuario no habia aprobado
      previamente esos permisos en Azure. Se simplifico a solicitar solo

7.3 BLOQUEO VIRTUAL DE ARCHIVOS (VIRTUAL FILE LOCK)
--------------------------------------------------------------------------------

  Dado que no tenemos un sistema de locking real, usamos un sistema
  de bloqueo virtual basado en worklog.md.

  [A] COMO BLOQUEAR UN ARCHIVO
  Cuando una sesion va a trabajar en un archivo peligroso (seccion 7.2[C]),
  debe registrar en worklog.md:

    LOCK: [archivo] por [agente/sesion] desde [hora] — [motivo]
    Ejemplo: LOCK: src/contexts/AppContext.tsx por Sesion-A desde 14:30 — Fix filtrado tenant

  [B] COMO VERIFICAR SI UN ARCHIVO ESTA BLOQUEADO
  Antes de tocar un archivo peligroso, buscar en worklog.md:
    rg "LOCK:" worklog.md

  Si el archivo aparece bloqueado por otra sesion activa:
    1. NO MODIFICAR el archivo
    2. Esperar a que la otra sesion haga UNLOCK
    3. O coordinar con el usuario para saber si ya termino

  [C] COMO DESBLOQUEAR UN ARCHIVO
  Al terminar de trabajar en el archivo:
    1. Hacer commit + push
    2. Registrar en worklog.md:
       UNLOCK: [archivo] por [agente/sesion] a [hora] — commit [hash]
    Ejemplo: UNLOCK: src/contexts/AppContext.tsx por Sesion-A a 15:45 — commit abc1234

  [D] EXPIRACION AUTOMATICA DE LOCKS
  ├── Si un LOCK tiene mas de 2 horas sin UNLOCK, se considera expirado
  ├── La sesion que necesite el archivo puede tomarlo despues de verificar
  │   que la otra sesion ya no esta activa (revisando worklog.md completo)
  ├── Al tomar un lock expirado, registrar: TAKEOVER: [archivo] — lock expirado de [agente]
  └── Intentar hacer git pull antes de modificar para integrar cambios

7.5 ROLLBACK Y RECUPERACION DE EMERGENCIA
--------------------------------------------------------------------------------

  [A] SI LA APP SE ROMPE EN PRODUCCION
    1. Identificar el commit que causo el problema: git log --oneline -5
    2. Revertir el commit: git revert [hash]
    3. Push: git push origin main
    4. Vercel hara deploy automatico del fix
    5. Documentar en bitacora con severidad CRITICA
    6. Si git revert no funciona: git reset --hard [hash-estable]
       (ULTIMO RECURSO — solo si es absolutamente necesario)

  [B] SI SE PIERDE CODIGO
    1. Buscar commits recientes: git reflog
    2. Restaurar desde un commit: git checkout [hash] -- [archivo]
    3. O restaurar desde tag: git checkout [tag] -- [archivo]
    4. Hacer commit del archivo restaurado
    5. Documentar en bitacora

  [C] SI SE PIERDEN DATOS DE FIRESTORE
    1. Verificar si hay backup en BackupScreen (v50)
    2. Restaurar desde backup si existe
    3. Si no hay backup: documentar exactamente que se perdio
    4. Informar al usuario inmediatamente
    5. NO intentar recuperar datos manualmente sin saber lo que se hace

  [D] SI DOS SESIONES SOBREESCRIBEN EL MISMO ARCHIVO
    1. NO usar git push --force (PROHIBIDO bajo cualquier circunstancia)
    2. Hacer git pull para obtener ambos cambios
    3. Resolver conflictos manualmente (git abrira conflict markers)
    4. Conservar TODOS los cambios de ambas sesiones
    5. Si no se puede resolver: guardar ambas versiones como archivos separados
       y pedir al usuario que decida que conservar
    6. Documentar el conflicto en worklog.md

7.8 ESCENARIOS DE ERROR COMUN Y COMO PREVENIRLOS
--------------------------------------------------------------------------------

  ESCENARIO 1: Dos sesiones modifican AppContext.tsx al mismo tiempo
  ├── Prevencion: LOCK en worklog.md antes de modificar
  ├── Si ya esta lockeado: esperar o pedir al usuario que coordine
  ├── Si ambos modificaron: hacer git pull y resolver conflictos
  └── NUNCA force push

  ESCENARIO 2: Una sesion elimina codigo que otra sesion necesita
  ├── Prevencion: antes de eliminar codigo, verificar con git log
  │   si fue agregado recientemente por otra sesion
  ├── Si no se sabe: preguntar al usuario antes de eliminar
  └── NUNCA eliminar codigo "porque parece innecesario"

  ESCENARIO 3: Una sesion rompe el build y otra no puede avanzar
  ├── Prevencion: cada sesion hace build local antes de push
  ├── Si el build esta roto: la sesion que lo rompio DEBE arreglarlo
  │   antes de hacer otra cosa
  ├── Si no puede arreglar: revertir su ultimo commit
  └── Documentar en bitacora con severidad ALTA

  ESCENARIO 4: Una sesion modifica tipos.ts y rompe toda la app
  ├── Prevencion: cambios en types.ts deben ser backwards-compatible
  ├── Si se necesita cambiar un tipo existente: crear un NUEVO tipo
  │   y migrar gradualmente (no romper el tipo actual)
  ├── Antes de cambiar tipos compartidos: verificar quien los usa
  │   rg "NombreTipo" src/ --type ts
  └── Hacer build despues de cambiar tipos para verificar

  ESCENARIO 5: Push a main gasta deploy de Vercel innecesariamente
  ├── Prevencion: trabajar en ramas (git checkout -b feature/nombre)
  ├── Solo mergear a main cuando la feature esta completa y testeada
  ├── Cuenta los deploys: cada push a main = 1 deploy
  └── Limite: 100 deploys/dia en plan gratuito de Vercel

  ESCENARIO 6: Una sesion sobreescribe el instructivo con version vieja
  ├── Prevencion: siempre git pull antes de modificar instructivo
  ├── El instructivo SOLO debe crecer (agregar al final)
  ├── NUNCA eliminar secciones del instructivo
  └── Si hay conflicto: conservar AMBAS versiones y el usuario consolida
